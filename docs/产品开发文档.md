## 1. 项目概述

### 1.1 产品定位

面向中医爱好者与中医从业者的**中医治疗方案查询与分享平台**。用户可以通过本平台查询各类疾病的治疗方案（针灸、中草药、拔罐、推拿等），也可以上传自己的中医资料，经 AI 整理加人工审核后入库，实现中医知识的共建共享。

### 1.2 核心价值

| 用户角色 | 价值 |
|---------|------|
| 中医爱好者 | 查到经过审核的可靠治疗方案，学习中医知识 |
| 中医从业者 | 交流临床经验，分享验方，获得收益和积分 |
| 平台 | 构建高质量的中医治疗方案数据库 |

### 1.3 核心数据流

用户上传内容 → AI 自动处理（提取/归类/结构化） → 人工审核 → 入库发布 → 前端可见 → 用户购买 → 查看完整内容 → 使用后提交疗效反馈 → 获得积分（可兑换）

## 2. 核心功能

### 2.1 治疗方案查询
- 按疾病名称搜索
- 按治疗类型筛选（针灸/拔罐/放血/推拿/方剂内服/方剂外用/食疗药膳等）
- 按疾病分类浏览（内科/外科/妇科/儿科/骨伤科/五官科/皮肤科等）
- 查看治疗方案详情（含预览 + 付费解锁完整内容）
- 查看各方案的疗效反馈汇总（评分、有效率统计、用户评价列表）

### 2.2 用户上传（一键式）
- 用户只需把内容放入上传框，无需填写任何字段
- 支持：直接粘贴文字、拖入 PDF/图片、拍照上传
- 多文件批量上传
- 上传后自动进入 AI 处理流程

### 2.3 AI 自动处理
- 文件解析（OCR 识别图片/PDF 文字）
- 调用大模型进行结构化提取（标题、疾病归属、类型、内容、预览、方剂明细等）
- 自动匹配或创建疾病条目
- 处理结果进入待审核池

### 2.4 人工审核
- 管理员对照原始内容与 AI 结果
- 可编辑修改 AI 生成的所有字段
- 审核通过 → 写入主表，前端可见
- 审核驳回 → 填写理由，通知上传者

### 2.5 疗效反馈
- 已购买用户可对治疗方案提交反馈
- 反馈内容：评分（1-5星）、疗效情况（有效/无效/部分有效）、文字描述、疗程天数、图片（如舌苔照）
- 每个治疗方案聚合显示反馈统计（有效率、平均评分、用户评价列表）

### 2.6 积分体系
- 提交反馈 → 获得积分
- 分享治疗方案 → 获得积分
- 他人通过分享链接购买 → 获得积分奖励
- 优质反馈被推荐 → 额外积分奖励
- 积分可兑换：付费内容访问权限、实体商品（艾条/刮痧板等）

### 2.7 分享功能
- 生成分享链接（带分享者追踪参数）
- 生成分享海报（含二维码）
- 追踪点击次数和转化次数

## 4. 核心业务流程

### 4.1 内容上传到发布全流程

用户上传 → AI 处理层（文本提取/PDF解析/OCR）→ 调用大模型进行结构化提取 → 待审核池（status=pending）→ 管理员审核页面 → 通过 → 写入主表并发布 → 前端可见

审核驳回时：填写理由 → 通知上传者 → uploads 表 status=rejected

### 4.2 付费购买流程

用户浏览治疗方案 → 查看免费预览 → 点击付费 → 调起支付宝/微信支付 → 支付成功 → 后端回调标记已购买 + 创建购买记录 + 触发分账 → 用户可查看完整内容（无限次）

### 4.3 反馈流程

用户已购买 → 使用治疗方案 → 进入详情页点击提交反馈 → 填写评分/疗效/描述/天数/图片 → 提交 → 获得50积分 → 更新该方案的评分和有效率统计

## 5. 数据库设计

### 5.1 用户与账户

**表名：users**
字段：id, phone(唯一), username, password_hash, role(user/reviewer/admin), avatar, balance(可提现余额), points(积分), total_earnings(累计收益), status, created_at, updated_at

**表名：points_log**
字段：id, user_id, amount(正/负), balance_after(变动后余额), source_type(feedback/share/share_purchase/exchange/admin_adjust), source_id, note, created_at

**表名：withdrawals**
字段：id, user_id, amount, fee, real_amount(实际到账), account_info(JSON), status(pending/approved/rejected/completed), admin_id, remark, created_at, processed_at

### 5.2 疾病与分类

**表名：diseases**
字段：id, name, pinyin(拼音), category_id, keywords(JSON用于AI归类), description, icon, sort_order, status, created_at

**表名：disease_categories**
字段：id, name(内科/外科/妇科/儿科等), parent_id, sort_order, created_at

### 5.3 治疗类型

**表名：treatment_types**
字段：id, name(针灸/拔罐/放血/推拿等), icon, sort_order, status, created_at

### 5.4 上传与审核（核心枢纽表）

**表名：uploads**
- id, user_id
- raw_type(text/image/pdf), raw_content(LONGTEXT), raw_file_paths(JSON)
- ai_result(JSON): AI的完整返回结果
- reviewed_title, reviewed_disease_id, reviewed_type_ids(JSON), reviewed_preview(TEXT), reviewed_full_content(LONGTEXT), reviewed_formulas(JSON), reviewed_acupuncture(JSON)
- status(processing/pending/approved/rejected)
- reject_reason(TEXT), reviewer_id, reviewed_at, created_at, updated_at

### 5.5 治疗方案

**表名：treatments**
字段：id, upload_id, disease_id, type_ids(JSON), title, preview_content(TEXT), full_content(LONGTEXT), price, avg_rating, total_ratings, effectiveness_rate, total_feedbacks, share_count, purchase_count, uploader_id, status, created_at, updated_at

**表名：purchases**
字段：id, user_id, treatment_id, amount, platform_fee(支付手续费), uploader_share(上传者分成), platform_share(平台分成), payment_method, payment_id, status(paid/refunded), created_at

### 5.6 疗效反馈

**表名：feedbacks**
字段：id, treatment_id, user_id, rating(1-5), effectiveness(effective/ineffective/partial), content(TEXT), course_days(使用天数), images(JSON), points_awarded(默认50), is_recommended(优质反馈标识), status, created_at

### 5.7 分享追踪

**表名：shares**
字段：id, user_id, treatment_id, share_type(link/image/poster), target(wechat_session/wechat_timeline/copy_link/other), unique_code(VARCHAR 32 UNIQUE), click_count, conversion_count(带来的购买数), points_awarded, created_at, updated_at

### 5.8 积分兑换

**表名：exchange_rules**
字段：id, name, description(TEXT), type(content/physical/vip), points_required, target_id, stock(库存,-1不限), image, status, created_at

**表名：exchanges**
字段：id, user_id, rule_id, points_cost, status(completed/cancelled), created_at

### 5.9 表关系概览

- users → uploads (用户上传内容)
- users → treatments(uploader_id) (上传者拥有治疗方案)
- users → purchases (用户购买记录)
- users → feedbacks (用户提交反馈)
- users → points_log (用户积分流水)
- users → shares (用户分享记录)
- users → exchanges (用户兑换记录)
- users → withdrawals (用户提现记录)
- users → uploads(reviewer_id) (审核人)
- uploads → treatments (上传审核通过后生成治疗方案)
- diseases → treatments (疾病下挂多个治疗方案)
- treatments → feedbacks (治疗方案有多个反馈)
- treatments → purchases (治疗方案有多个购买记录)
- treatments → shares (治疗方案有多个分享)

## 6. AI 自动处理管线

### 6.1 处理流程

**步骤1 - 内容解析**：纯文本直接使用 / PDF调用解析库提取文本 / 图片调用OCR提取文字

**步骤2 - 调用大模型结构化提取**：系统角色设定为"专业中医数据整理助手"，提取标题/疾病名称/治疗类型/预览/完整内容/方剂明细/针灸方案，以JSON格式返回

**步骤3 - 疾病匹配与归类**：将AI提取的疾病名称与数据库diseases表进行名称/关键词匹配，匹配到则关联疾病ID，未匹配则新建疾病条目（标记待完善）

最终保存到 uploads 表（status=pending），等待人工审核

### 6.2 AI 调用成本估算（GPT-4o-mini）

| 内容类型 | Token消耗 | 单次成本 |
|---------|-----------|---------|
| 短文字（几百字） | ~500 tokens | ~0.001元 |
| 长文字（几千字） | ~3000 tokens | ~0.006元 |
| PDF/图片+OCR | ~2000 tokens | ~0.004元 |

日均100条处理量月成本约15-30元。

### 6.3 异常处理

- AI返回格式不对：重试2次，失败则人工介入
- OCR识别失败：提示用户尝试文字输入
- 内容非中医相关：AI标记low_quality，审核时优先驳回
- AI提取为空：保留原始内容，审核员手动处理

## 7. 审核机制

### 7.1 审核标准

| 审核项 | 通过标准 |
|--------|---------|
| 标题 | 准确概括内容，无夸大宣传 |
| 疾病归属 | 与内容所述疾病一致 |
| 治疗类型 | 与实际治疗方法匹配 |
| 预览内容 | 信息准确，吸引人，不泄露核心信息 |
| 完整内容 | 方剂有剂量，穴位有手法，信息完整可操作 |
| 安全性 | 无不适合公开的危险内容，含毒性药材需标注 |

### 7.2 驳回管理
- 驳回时必须选择或填写理由
- 系统通知上传者驳回原因
- 上传者可参考驳回理由修改后重新上传
- 累计3次被驳回的内容标记为违规，限制该用户上传权限

## 11. 盈利模式与分账体系

### 11.1 收入来源

| 来源 | 说明 |
|------|------|
| 单次付费查看 | 预览免费，查看完整内容付费（3-30元） |
| 会员订阅（后续） | 月/季/年卡，全站内容免费看 |
| 积分兑换未消耗（后续） | 积分兑换商品差价 |
| 广告收入（后续） | 适量非侵入式广告 |

### 11.2 分账规则

用户支付10元 → 扣除支付手续费约0.6%（0.06元）→ 可分配金额9.94元 → 上传者分成50%（4.97元）→ 平台分成50%（4.97元）

### 11.3 提现规则
- 最低提现金额：50元
- 提现手续费：超过每月2次后收取1%
- 提现方式：支付宝/微信转账
- 审核周期：提交后1-3个工作日处理

### 13.3 关键页面说明

**上传页（核心亮点）**
- 一个空白大框，提示语："粘贴中医内容，或拖入文件/图片"
- 支持 multipart/form-data 上传
- 上传后显示处理进度条（processing → pending）
- 处理完成后可跳转到上传记录页查看审核状态

**治疗方案详情页**
- 顶部：标题 + 类型标签 + 评分 + 有效率
- 中部：免费预览内容
- 分隔线 + 付费按钮（或已购提示）
- 底部：疗效反馈聚合区

**审核页（仅管理员）**
- 分栏布局：原始内容 vs AI 建议
- 所有字段可编辑
- 支持左右滑动切换上一条/下一条待审核内容

## 14. 技术栈

### 14.1 前端
| 技术 | 选型 | 说明 |
|------|------|------|
| 框架 | Flutter 3.x | 跨平台移动端 |
| 状态管理 | Provider 或 Riverpod | 推荐 Riverpod |
| 网络请求 | Dio | HTTP 封装 |
| 路由 | go_router | 声明式路由 |
| 本地存储 | shared_preferences + sqflite | 缓存 |

### 14.2 后端
| 技术 | 选型 | 说明 |
|------|------|------|
| 框架 | Laravel 11.x | PHP 全栈框架 |
| 数据库 | MySQL 8.x | 关系型数据库 |
| 缓存/队列 | Redis | 队列任务 + 缓存 |
| API 认证 | JWT (laravel-sanctum) | 无状态认证 |
| 队列驱动 | Laravel Queue + Redis | 异步处理 AI 任务 |
| 支付 | 支付宝/微信 SDK | 官方 SDK |

### 14.3 AI 服务
| 服务 | 用途 |
